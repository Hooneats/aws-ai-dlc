# AIDLC 워크샵 리뷰 — 4시간 만에 테이블오더 시스템을 만들다

## 시작은 기획부터

테이블오더 시스템. 식당에서 QR 찍고 메뉴 고르고 주문하는, 우리가 매일 쓰는 그 시스템을 직접 만들어보기로 했다.

처음에는 "뭘 만들 것인가"부터 정리했다. 메뉴 관리, 주문 접수, 테이블 매핑, 주방 알림까지 — 기능 목록을 나열하고, 우선순위를 잡고, 어떤 흐름으로 사용자가 움직일지 설계하는 과정을 거쳤다. AIDLC의 Inception Phase를 따라가면서 요구사항을 분석하고 설계를 잡아나갔는데, AI가 질문을 던지고 구조를 제안하는 방식이 생각보다 체계적이었다. 혼자였으면 화이트보드 앞에서 한참 고민했을 내용들이 대화 몇 번 만에 정리됐다.

## 열정이 과했다

솔직히 말하면, 초반에 욕심을 너무 부렸다.

기획 단계에서 "이것도 넣자, 저것도 되면 좋겠다"를 반복하다 보니 스코프가 슬금슬금 커졌다. 4시간이라는 제한 시간 안에서 기획에 공을 들인 만큼, 뒤쪽 시간이 빠듯해졌다. 특히 코드를 다 작성하고 나서 디버깅할 시간이 턱없이 부족했던 게 가장 아쉬웠다. 동작은 하는데 엣지 케이스를 잡을 여유가 없었고, "조금만 더 다듬으면 되는데"라는 생각이 계속 남았다.

다음에 비슷한 시간 제약이 있다면, 기획 단계에서 과감하게 쳐내는 연습이 필요하다는 걸 몸으로 배웠다. MVP라는 단어를 머리로는 알고 있었지만, 실제로 시간에 쫓기면서 체감한 건 이번이 처음이었다.

## 코드 작업, 생각보다 오래 걸렸다

AI가 코드를 생성해주니까 금방 끝날 줄 알았다. 그런데 실제로는 그렇지 않았다.

생성된 코드를 이해하고, 프로젝트 구조에 맞게 조정하고, 컴포넌트 간 연결을 확인하는 데 시간이 꽤 들었다. AI가 뼈대를 잡아주는 건 확실히 빠르지만, 그걸 실제 동작하는 시스템으로 엮어내는 건 결국 사람의 몫이었다. 코드 생성 속도와 코드 완성 속도는 다른 문제라는 걸 느꼈다.

## 다른 분야의 사람들이 하나의 시스템을 만든다는 것

이번 워크샵에서 가장 놀라웠던 건, 개발 배경이 아닌 사람들도 함께 하나의 프로젝트를 만들어냈다는 점이다.

기획자, 디자이너, PM — 각자 다른 언어로 일하던 사람들이 AI를 매개로 같은 시스템을 설계하고 구현하는 과정에 참여했다. 예전이었으면 "이건 개발자한테 넘겨야 해"로 끝났을 대화가, "그러면 이 부분은 이렇게 바꿔볼까요?"로 이어졌다. AI가 기술적 장벽을 낮춰주니까, 논의의 깊이가 달라졌다.

## DDD의 유비쿼터스 언어, 그게 실현되는 순간

개인적으로 가장 감명받은 부분이 있다.

DDD(Domain-Driven Design)에서 유비쿼터스 언어(Ubiquitous Language)를 정의하려면, 원래는 개발자, 기획자, 도메인 전문가, 운영팀 — 유관 부서가 전부 모여서 용어를 합의해야 한다. "주문"이 뭔지, "테이블 상태"가 뭘 의미하는지, 모두가 같은 단어를 같은 뜻으로 쓰도록 맞추는 작업이다. 현실에서는 이게 몇 주, 길면 몇 달이 걸리는 일이다.

그런데 이번 워크샵에서는 그 과정이 하나의 언어, 하나의 도구 안에서 자연스럽게 이루어졌다. AI와 대화하면서 도메인 용어를 정의하고, 그 용어가 그대로 요구사항에 반영되고, 코드에까지 일관되게 흘러갔다. 기획 문서의 "메뉴 카테고리"가 코드의 `MenuCategory`로 자연스럽게 연결되는 경험. 이건 단순히 편리한 게 아니라, DDD가 꿈꿨던 이상에 한 발짝 다가간 느낌이었다.

## 마무리하며

4시간은 짧았다. 아쉬움도 남았다. 하지만 그 짧은 시간 안에 기획부터 설계, 구현까지 하나의 시스템을 만들어본 경험은 꽤 강렬했다.

AI가 개발자를 대체하는 게 아니라, 개발이라는 행위의 진입장벽을 낮추고, 서로 다른 역할의 사람들이 같은 테이블에서 같은 언어로 대화할 수 있게 해준다는 것. 그게 이번 워크샵에서 얻은 가장 큰 인사이트였다.

> 완벽하진 않았지만, 그래서 더 기억에 남는 4시간이었다.
