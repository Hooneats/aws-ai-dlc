# 산출물 교차 검증 규칙 (Cross-Validation Rule)

## 목적
모든 산출물 파일 생성 전에 기존 산출물과의 일관성을 검증하여 누락/불일치를 방지한다.

## 적용 시점
**MANDATORY**: 아래 상황에서 반드시 실행한다:
1. 새로운 산출물 파일을 생성하기 전
2. 기존 산출물 파일을 수정한 후
3. 각 단계(Stage) 완료 시

## 검증 항목

### 1. 요구사항 ↔ 유저 스토리 일관성
- [ ] 모든 FR(기능 요구사항)에 대응하는 유저 스토리가 존재하는가
- [ ] 유저 스토리의 내용이 요구사항과 일치하는가
- [ ] 요구사항 변경 시 유저 스토리도 함께 업데이트되었는가

### 2. 요구사항 ↔ 컴포넌트 일관성
- [ ] 모든 FR을 처리하는 컴포넌트가 식별되었는가
- [ ] 데이터 모델의 엔티티가 컴포넌트에 반영되었는가

### 3. 컴포넌트 ↔ 메서드 일관성
- [ ] 모든 컴포넌트의 책임에 대응하는 메서드가 정의되었는가
- [ ] 서비스 플로우에서 호출하는 메서드가 메서드 시그니처에 존재하는가
- [ ] 메서드 입출력 타입이 데이터 모델과 일치하는가

### 4. 서비스 플로우 ↔ 의존성 일관성
- [ ] 서비스 플로우에서 사용하는 모듈 간 호출이 의존성 매트릭스에 반영되었는가
- [ ] 의존성 매트릭스에 없는 모듈 간 호출이 서비스 플로우에 존재하지 않는가

### 5. 문서 내부 일관성
- [ ] 동일 항목의 ID/이름이 문서 간 일치하는가 (예: FR-C05 중복 없음)
- [ ] 변경된 설계 결정이 모든 관련 문서에 반영되었는가

### 6. 데이터 모델 완전성
- [ ] 모든 엔티티에 필요한 필드가 정의되었는가
- [ ] 엔티티 간 관계가 명확한가
- [ ] 특수 동작을 위한 구분 필드가 존재하는가 (예: isServiceCategory)

## 실행 방법
1. 관련 산출물 파일들을 모두 로드
2. 위 검증 항목을 순서대로 체크
3. 누락/불일치 발견 시 사용자에게 보고
4. 사용자 확인 후 수정 진행
5. 수정 후 재검증

## 보고 형식
```
### 🔴 누락된 부분
- [번호]. [설명] - [영향받는 파일]

### 🟡 경미한 불일치
- [번호]. [설명] - [영향받는 파일]

### ✅ 검증 통과
- 모든 검증 항목 통과
```
